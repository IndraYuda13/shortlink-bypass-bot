# Probe notes: exe.io / exeygo deep pass

Date: 2026-04-24
Scope: `https://exe.io/vkRI1` -> expected `https://google.com`
Method: safe HTTP only using project `.venv` `curl_cffi` Chrome impersonation plus static HTML/JS inspection. No browser automation, no captcha solving, no code edits.

## Summary

`exe.io/vkRI1` is a two-stage CakePHP shortlink flow hosted on `exeygo.com`. The first form (`before-captcha`) is replayable with hidden fields and advances to the second form (`link-view`). The second form is the active captcha/timer gate; replaying it without a valid captcha token, or with fake `cf-turnstile-response` / `g-recaptcha-response`, resets back to `/vkRI1` and does not reveal the final target.

The expected target `https://google.com` is **not proven as embedded or derivable** from the fetched gate HTML. The only `google.com` occurrence relevant to a destination-looking URL is part of a random `document.referrer` spoofing snippet alongside YouTube/Reddit, so it must not be treated as final-target evidence.

## Embedded target check

The only destination-looking `google.com` snippet in both gate pages is:

```js
const os = ["https://google.com/", "https://youtube.com/", "https://reddit.com/"];
Object.defineProperty(document, "referrer", {
  get: function () {
    return os[Math.floor(Math.random() * os.length)];
  }
});
```

This is a referrer randomization/anti-analysis snippet, not a final redirect assignment. It appears with multiple alternatives and should be treated as noise. Other `google` occurrences are Google Analytics / signup copy, not target evidence.

No base64 token, hidden field, form action, cookie, inline script assignment, or fetched static script exposed `https://google.com` as the final target.

## Handler recommendation

Add an `exe.io` / `exeygo.com` handler only as a gated mapper until captcha solving is available:

1. Use `curl_cffi` Chrome impersonation.
2. `GET exe.io/<alias>` with `allow_redirects=False`; expect `302` to `https://exeygo.com/<alias>`.
3. `GET exeygo.com/<alias>`; parse `form#before-captcha` hidden fields and `app_vars`.
4. `POST form#before-captcha` exactly; parse `form#link-view` hidden fields.
5. Wait `counter_value` seconds (`6` in this sample, starts on `DOMContentLoaded`).
6. Submit `form#link-view` only with a valid captcha solver token for the active `captcha_type` (`turnstile` observed now; keep reCAPTCHA fallback because template exposes both keys).
7. Claim the final target only from the post-captcha redirect/body. Do **not** use the `document.referrer` randomizer as target evidence.

Recommended engine result before solver integration: `CAPTCHA_REQUIRED` / `EXE_GATE_MAPPED`, with facts for form sequence, sitekeys, timer, and blocker `valid Turnstile/reCAPTCHA token required before final target can be proven`.
